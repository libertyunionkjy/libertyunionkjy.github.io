# Git Git和代码托管中心 **代码托管中心：维护远程库** * 局域网环境下：gitlab * 外网环境下：github、码云 **本地库和远程库：** 团队内协作——本地库： ![image-20200823120419336](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200823120808.png) 跨团队协作——远程库： ![image-20200823121200896](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200823121201.png) ## Git命令行操作 ### Git本地库初始化：`git init` ![image-20200823122221265](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200823122221.png) .git中存放的是**本地库相关**的子目录和文件 ### 设置签名 * 形式 * 用户名:Lemon * Email地址:xxxxx@gmail.com * 作用：区分不同开发人员身份 * 辨析：这里设置的签名和登录远程库（代码托管中心）的账号、密码没有任何关系 * 命令 * 项目级别/仓库级别：仅在当前本地库范围生效 * `git config user.name Lemon` * `git config user.email xxxx@gmail.com` * 信息保存位置：.git/config ![image-20200823123323940](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200823123324.png) * 系统用户级别：登录当前操作系统的用户范围 * `git config --global user.name Lemon` * `git config --global user.email xxxx@gmail.com` * 保存位置：用户目录下的.gitconfig文件 ![image-20200823123620102](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200823123620.png) * 优先级：就近原则，二者都有时，项目级别>系统用户级别 * 二者都没有不允许 ### 基本操作 > 新建的文件必须要add一下再commit > > 修改的文件可以直接commit，但是不能够撤销了 > 多屏显示的控制方式：空格向下翻页、b向上翻页、q退出 ![image-20200824011317501](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200824011949.png) > HEAD{移动到当前版本需要多少步数} #### **前进后退** 1. :star:基于索引值[推荐] ![image-20200824013924266](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200824013926.png) 2. 基于`^`：只能往后退 3. 基于`~`：只能后退 #### **reset参数对比** 1. --soft:只碰本地库的HEAD指针 ![image-20200824095959578](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200824095959.png) 2. --mix：修改本地库的HEAD指针和重置暂存区 ![image-20200824100336981](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200824100337.png) 3. -- hard：修改本地库的HEAD指针，重置暂存区、工作区 #### 删除文件后找回 前提：删除前文件存在时的状态提交到了本地库 操作：`git reset --hard [指针位置`] **永久删除文件后找回** `git reset --hard ` **添加到暂存区的删除文件找回** `git reset --hard HEAD` #### git diff ![0A44FE6F4EB0A0EC577F17855F53D4D3](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200824105230.png) [reference](https://www.jianshu.com/p/80542dc3164e) ##### 工作目录 vs 暂存区 意义：查看文件在工作目录与暂存区的差别。如果还没 add 进暂存区，则查看文件自身修改前后的差别。也可查看和另一分支的区别。 ##### 暂存区 vs Git仓库 意义：表示查看已经 add 进暂存区但是尚未 commit 的内容同最新一次 commit 时的内容的差异。 也可以指定仓库版本： ##### 工作目录 vs Git仓库 意义：查看工作目录同Git仓库指定 commit 的内容的差异。 ``=`HEAD` 时：查看工作目录同最近一次 commit 的内容的差异。 ##### Git仓库 vs Git仓库 意义：Git仓库任意两次 commit 之间的差别。 ### 分支 ![image-20200824123430188](https://cdn.jsdelivr.net/gh/libertyunionkjy/jykeimg/images/20200824130817.png) **合并分支** * 切换到接受修改的分支 * 执行merge命令 * `git merge `； **合并分支的时候产生冲突** 1. 冲突的表现 2. 冲突的解决 1. 编辑文件，删除特殊符号 2. 修改文件，保存 3. `git add ` 4. `git commit -m "commit message"`(不能带文件名) ## Git基本原理 SVN：增量式 Git：基于快照 提交：`git commit` 新建 分支：`git branch ` 切换分支：`git checkout ` 新建分支并且切换到那个分支：`git checkout -b ` 合并分支： 1. `git merge `； 2. 得到更线性的提交，把当前分支合并到`branch_name上`，`git rebase `； 分离HEAD：`git checkout <某次提交的哈希值>`，然后HEAD就会指向这个哈希的提交。 查看提交记录的哈希值：`git log` **相对引用** `^` * `git checkout master^`、HEAD会指向master的父节点； * `git checkout master^^`、HEAD会指向master的父节点的父节点； * master可以换成任何一个节点名称 `~` * `git checkout HEAD~4`：向后移动四步 * ~后面可以不跟参数，此时与^无异 **强制移动分支** * `git branch -f master HEAD~3`：上面的命令会将 master 分支强制指向 HEAD 的第 3 级父提交。 > 相对引用为我们提供了一种简洁的引用提交记录 `C1` 的方式， 而 `-f` 则容许我们将分支强制移动到那个位置。 **撤销变更** `git reset` 通过把分支记录回退几个提交记录来实现撤销改动。你可以将这想象成“改写历史”。`git reset` 向上移动分支，原来指向的提交记录就跟从来没有提交过一样。 虽然在本地分支中使用 `git reset` 很方便，但是这种“改写历史”的方法对大家一起使用的远程分支是无效的。 `git revert`：新生成一个提交，这个提交正好是用来撤销更改的。 将一些提交复制到当前位置下面：`git cherry-pick <提交号>...` **交互式rebase：`git rebase -i`** [git book](http://gitbook.liuhui998.com/4_3.html) # Github Pull Request(PR)：如果我fork了别人的项目，自己做了改进到自己的仓库里面，但我同时希望我的改进可以加到被fork的人的项目中（项目创始人），我就可以发起一个PR ## git remote ## git push ## git pull * pull 相当于fetch+merge * `git fetch <远程库地址别名> <远程分支名>` * `git merge <远程库地址别名/远程分支名>` ## 多人协作 ### git clone 1. 完整的把远程库下载到本地 2. 创建origin远程地址别名 3. 初始化本地库 ### 冲突解决 * 要点 * 如果不是基于远程库所做的修改，不能推送，必须先拉取 * 拉取下来后如果进入冲突状态，按照冲突操作解决 ### 邀请成员加入团队 [github文档](https://docs.github.com/cn/github/setting-up-and-managing-organizations-and-teams/adding-organization-members-to-a-team) ### 团队外部人员协同开发 外部人员： 1. fork项目 2. 编写代码，提交到自己的远程库 3. 提交PR 项目拥有者： 1. 审查 2. 确认merge 3. pull到本地库 ### SSH免密登录 1. 把自己的公钥放到github上 2. `git remote add <别名> <仓库的ssh地址>`
